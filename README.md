# Roket-factory

Учебный backend-проект по курсу «Микросервисы, как в BigTech 2.0». Репозиторий демонстрирует микросервисную архитектуру для сервиса заказа компонентов ракетной техники: отдельный HTTP API для заказов, gRPC-сервисы каталога деталей и оплаты, общие контракты, unit-тесты и инфраструктуру для линтинга, генерации кода и проверки покрытия.

На текущем этапе проект находится в разработке. Основная реализация сервисов расположена в каталоге `boilerplates/`.

## Что реализовано

- `order` сервис с HTTP API для работы с заказами:
  - создание заказа;
  - получение заказа по UUID;
  - оплата заказа;
  - отмена заказа.
- `inventory` сервис на gRPC:
  - получение списка доступных деталей;
  - получение детали по UUID;
  - in-memory каталог с предзаполненными позициями.
- `payment` сервис на gRPC:
  - обработка оплаты заказа;
  - генерация UUID транзакции;
  - выделенный сервисный слой под дальнейшее расширение бизнес-логики.
- Общий слой контрактов:
  - OpenAPI-спецификация для `order`;
  - Protobuf-контракты для `inventory` и `payment`;
  - сгенерированный код клиентов и серверных интерфейсов.
- Архитектурное разделение по слоям:
  - `api` - транспортный слой;
  - `service` - бизнес-логика;
  - `repository` - доступ к данным;
  - `converter` / `model` - преобразования и доменные модели.
- Тестовая инфраструктура:
  - unit-тесты для `api` и `service`;
  - моки через `mockery`;
  - task-команды для расчета покрытия и запуска проверок.

## Архитектура

Проект состоит из нескольких Go-модулей, объединенных через `go.work`:

- `boilerplates/order` - HTTP-сервис заказов.
- `boilerplates/inventory` - gRPC-сервис каталога деталей.
- `boilerplates/payment` - gRPC-сервис оплаты.
- `boilerplates/shared` - общие protobuf/OpenAPI контракты и сгенерированный код.

### Взаимодействие сервисов

- `order` поднимает HTTP API на `localhost:8080`.
- `inventory` поднимает gRPC API на `localhost:50051`.
- `payment` поднимает gRPC API на `localhost:50052`.
- Контракты `order` описаны через OpenAPI.
- Контракты `inventory` и `payment` описаны через Protobuf/gRPC.

```powershell

go run .\cmd
```

```powershell
cd boilerplates\payment
go run .\cmd
```

```powershell
cd boilerplates\order
go run .\cmd
```

## Проверки и качество

В проекте подготовлены task-команды для:

- форматирования кода;
- линтинга;
- генерации protobuf-кода;
- генерации моков;
- запуска тестов;
- расчета test coverage.

Примеры:

```powershell
cd boilerplates
task lint
task test-coverage
task proto:gen
```

Локально подтверждено:

- `order` тесты проходят;
- `inventory` тесты проходят;
- `payment` тесты проходят;
- при точечных локальных запусках в sandbox может потребоваться свой `GOCACHE`, так как системный cache на Windows иногда недоступен по правам.

## Что показывает проект

Проект демонстрирует, что в рамках pet/учебной разработки реализованы:

- разбиение монолита на отдельные сервисы;
- проектирование API-контрактов;
- работа с HTTP и gRPC одновременно;
- выделение слоев приложения по принципам чистой архитектуры;
- покрытие ключевых use case unit-тестами;
- подготовка репозитория под CI/CD и автоматические проверки.

## Статус

Сейчас `Roket-factory` находится в стадии развития. Базовый каркас микросервисной системы уже реализован, а следующим шагом может быть добавление постоянного хранилища, docker-compose окружения, межсервисной бизнес-логики и интеграционных тестов.
